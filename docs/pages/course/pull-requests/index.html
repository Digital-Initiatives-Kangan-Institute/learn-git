Pull Requests

Reviewing

When merging pull requests into your repository, it is important to review the proposed changes before accepting them. A pull request review helps ensure that the code being added is correct, safe, and follows the standards of the project. It also helps prevent mistakes from being introduced into the main codebase.

When reviewing a pull request, you should check the following:

  • The changes match what is intended

    • Review the files to ensure the pull request only contains the changes needed for the task and no accidental or unrelated edits.

  • The changes are targeting the correct branch

    • Check that the pull request is being merged into the correct branch (such as main or develop) to avoid disrupting the workflow.

  • The code meets team quality standards

    • Ensure the code follows the team’s coding standards, is readable, and avoids unnecessary complexity.

  • The changes do not introduce bugs

    • Consider whether the update could break existing functionality and confirm the project still works as expected.

  • Documentation or comments are included if needed

    • Make sure any complex logic or new features are explained so other developers can understand and maintain the code.

Taking a few minutes to properly review a pull request helps maintain code quality, stability, and collaboration within the team. It also provides an opportunity to give constructive feedback and help other developers improve their work.
